# Шаблон Story для Jira

Используйте этот шаблон для каждой Story в Jira. Он помогает формулировать задачи так, чтобы их можно было реализовать, проверить и принять без неоднозначностей.

## Шаблон карточки Story

**Название**  
`[Область] Короткий бизнес-результат`

**User Story**  
Как `<роль>`, я хочу `<возможность>`, чтобы `<бизнес-ценность>`.

**Бизнес-контекст**  
- Почему это важно сейчас.
- На какой KPI/метрику это влияет.
- Связанный Epic/инициатива.

**Scope (границы)**  
- Входит в задачу:
- Не входит в задачу:

**Зависимости**  
- Backend/API:
- Frontend/UI:
- Внешние сервисы:

**Дизайн/ссылки**  
- Figma:
- API-контракт:
- Связанные задачи:

## Критерии приемки (Acceptance Criteria)

Пишите критерии в формате Дано/Когда/Тогда. Минимум 3 критерия.

### AC-1
**Дано:** `<исходный контекст>`  
**Когда:** `<действие>`  
**Тогда:** `<ожидаемый результат>`

### AC-2
**Дано:** `<исходный контекст>`  
**Когда:** `<действие>`  
**Тогда:** `<ожидаемый результат>`

### AC-3
**Дано:** `<исходный контекст>`  
**Когда:** `<действие>`  
**Тогда:** `<ожидаемый результат>`

Опционально добавляйте нефункциональные критерии:
- Производительность: время ответа, допустимый уровень ошибок
- Безопасность: проверки доступа, валидация данных
- Локализация: перевод RU/KK в требуемых местах

---

## Definition of Done (DoD)

Story считается завершенной только если выполнены все пункты:

1. Функциональность соответствует всем критериям приемки.
2. Код влит в основную ветку через PR и ревью.
3. По Story нет открытых багов уровня Critical/High.
4. Изменения API задокументированы (OpenAPI/README/changelog по необходимости).
5. Тесты добавлены или обновлены:
   - Unit-тесты для бизнес-логики
   - Integration-тесты для API-потоков (если затронут backend)
   - UI/interaction тесты для критичного поведения (если затронут frontend)
6. Ручной QA-чеклист пройден для happy path и ключевых edge cases.
7. Для нового потока добавлены логирование и обработка ошибок.
8. Выполнены базовые проверки безопасности (валидация входа, authz/authn, работа с чувствительными данными).
9. Изменение добавлено в release notes (или заметки к демо спринта).
10. Story принята Product Owner (или ответственным стейкхолдером).

---

## Быстрый QA-чеклист

- Happy path работает end-to-end.
- Невалидный ввод обрабатывается с понятным сообщением для пользователя.
- Состояния empty/loading/error отображаются корректно.
- Ролевой доступ применяется корректно.
- Нет регрессий в связанных сценариях.
