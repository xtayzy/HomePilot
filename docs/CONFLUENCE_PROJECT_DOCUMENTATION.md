## 1. Обзор проекта

**HomePilot** — платформа подписки на бытовые услуги: уборка, вынос мусора, закуп продуктов, выгул животных, мелкий бытовой ремонт и другие стандартные задачи по дому. Клиент выбирает тариф, указывает тип квартиры и расписание **слотов бытовой помощи**; в каждом слоте собирает нужные задачи из доступного списка. Платформа назначает исполнителя, ведёт календарь слотов/визитов, приём оплаты и фото-отчёты. Администратор управляет пулом исполнителей, расписанием и поддержкой.

| Параметр | Описание |
|----------|----------|
| **Продукт** | Подписка на слоты бытовой помощи (визиты) с фиксированной ценой за месяц |
| **Регион MVP** | Алматы (один город) |
| **Платформа** | Веб-приложение (адаптивное) |
| **Языки** | Русский, казахский (i18n) |
| **Валютa** | Тенге (KZT) |

**Ключевые принципы:** чёткий объём работ, закреплённый исполнитель, прозрачные тарифы, фото-отчёты после визита, поддержка в приложении.

## 2. Цели и границы MVP

**В рамках MVP:**
- Один город: **Алматы**
- Тарифы: Старт, Базовый, Оптимум, Комфорт, Премиум (4 или 8 визитов в месяц)
- Типы квартир: студия, 1-, 2-, 3-комнатная
- Тип уборки: лёгкая (поддерживающая) или полная
- Веб-приложение (адаптивное), без мобильных приложений

**Вне MVP (после запуска):**
- Мобильные приложения (iOS/Android)
- Второй город (Астана)
- Реферальная программа и промокоды
- Расширенные категории задач (visit_tasks, task_categories)

## 3. Роли пользователей

| Роль | Описание | Доступ |
|------|----------|--------|
| **Клиент (client)** | Покупатель подписки. Регистрируется, выбирает тариф, оплачивает, управляет визитами, смотрит отчёты. | Лендинг, тарифы, регистрация/вход, кабинет клиента (/dashboard/*) |
| **Исполнитель (executor)** | Выполняет визиты. Получает задания, отмечает выполнение по чек-листу, загружает фото. Регистрация только по приглашению. | Кабинет исполнителя (/executor/*) |
| **Администратор (admin)** | Сотрудник HomePilot. Управляет клиентами, подписками, визитами, тарифами, исполнителями. | Админ-панель (в разработке) |
| **Оператор поддержки (support)** | Отвечает на обращения клиентов и исполнителей. | Раздел поддержки в админке |

Роль передаётся в JWT и проверяется на бэкенде для каждого защищённого эндпоинта.

## 4. Продукт и пользовательские сценарии

### 4.1 Клиент (гость → подписчик)

1. **Лендинг** — описание сервиса, блок тарифов с ценой «от», CTA «Начать» / «Выбрать тариф».
2. **Тарифы** — выбор тарифа и типа квартиры → итоговая цена в месяц, кнопка «Оформить подписку».
3. **Регистрация / Вход** — email, пароль, имя, телефон; подтверждение email по 6-значному коду.
4. **Оформление подписки** — адрес (город, улица, дом, квартира, подъезд, этаж, код), удобные дни недели, окно времени (например 10:00–14:00), подтверждение оферты → переход к оплате.
5. **Оплата** — сумма за первый месяц; тестовый режим: ввод 6-значного кода из логов бэкенда после создания платежа.
6. **Личный кабинет** — дашборд (тариф, следующее списание, ближайший визит), мои визиты, настройка слотов, подписка и оплата, профиль.
7. **Визиты** — список/карточки визитов (дата, время, статус, исполнитель); детали визита, перенос (при соблюдении правил 24 ч), просмотр фото-отчёта после выполнения.
8. **Поддержка** — создание тикета, история обращений; «Есть претензия» по визиту (в разработке на фронте).

### 4.2 Исполнитель

1. **Вход** — по приглашению (тестовые логины создаются сидом).
2. **Дашборд** — визиты на сегодня и на неделю.
3. **Детали визита** — адрес, контакт, время, чек-лист по типу уборки.
4. **Выполнение** — «Начать визит», отметка пунктов чек-листа, загрузка фото по пунктам, «Завершить визит»; при неявке клиента — «Неявка» (no_show).

### 4.3 Админ (планируется)

Дашборд, клиенты, подписки, визиты (назначение исполнителя, перенос, отмена), исполнители (приглашение, зоны, статус), тарифы и цены, справочники, тикеты поддержки, платежи.

## 5. Архитектура системы

- **Монолит** с разделением модулей: users, subscriptions, visits, payments, support, notifications.
- Все запросы к API (кроме публичных) — с **JWT** в заголовке `Authorization: Bearer <token>`. Роль проверяется на каждом защищённом эндпоинте.
- Префикс API: **`/api/v1`**.

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   Frontend      │     │   Backend       │     │   PostgreSQL    │
│   (React/Vite)  │────▶│   (FastAPI)     │────▶│   (БД)          │
│   :3003         │     │   :8001         │     │   :15432        │
└─────────────────┘     └────────┬────────┘     └─────────────────┘
                                 │
                                 ├── /uploads (фото визитов, локально)
                                 ├── SMTP (email) или вывод кода в логи
                                 └── Webhook (платежи)
```

## 7. Модели данных

Основные сущности (SQLAlchemy). Полное описание полей — в [BACKEND_TZ_FASTAPI.md](BACKEND_TZ_FASTAPI.md).

| Сущность | Назначение |
|----------|------------|
| **User** | id, email, password_hash, role (client/executor/admin/support), name, phone, locale, email_verified_at, executor_status, photo_url, executor_invite_code |
| **City** | id, code, name_ru, name_kk, is_active |
| **ApartmentType** | id, code (studio, 1room, 2room, 3room), name_ru, name_kk, duration_light_min, duration_full_min |
| **Tariff** | id, code (start, basic, optimum, comfort, premium), name_ru, name_kk, cleaning_type (light/full), visits_per_month (4/8), has_linen, has_plants, has_ironing, is_active, sort_order |
| **TariffPrice** | tariff_id, apartment_type_id, price_month_kzt (уникальная пара тариф × тип квартиры) |
| **ChecklistTemplate** | cleaning_type, apartment_type_id (nullable), sort_order |
| **ChecklistItem** | template_id, title_ru, title_kk, sort_order |
| **Subscription** | user_id, tariff_id, apartment_type_id, city_id, адрес (address_*), preferred_days (JSON), time_slot_start/end, status (draft/active/paused/cancelled), started_at, ends_at, paused_until, executor_id, auto_renew |
| **Visit** | subscription_id, executor_id, scheduled_date, time_slot_start/end, status (scheduled/in_progress/completed/cancelled/no_show/rescheduled), completed_at |
| **VisitPhoto** | visit_id, checklist_item_id (nullable), file_path, uploaded_at |
| **VisitChecklistResult** | visit_id, checklist_item_id, done, photo_id |
| **Payment** | user_id, subscription_id, amount_kzt, currency, status (pending/completed/failed/refunded), external_id, paid_at |
| **SupportTicket** | user_id, visit_id (nullable), subject, status (open/in_progress/closed) |
| **SupportMessage** | ticket_id, author_id, author_role, body |
| **ExecutorZone** | executor_id, city_id, zone_name |
| **PaymentConfirmCode** / **EmailConfirmCode** | коды подтверждения (тестовые платежи и email) |

Статус подписки **draft** — создана, ожидает оплаты. После успешной оплаты — **active**, заполняются started_at/ends_at, создаются визиты на месяц по preferred_days.

## 11. Установка и запуск

### Вариант 1: Docker (рекомендуется)

```bash
cd backend
docker compose up --build
```

- **PostgreSQL:** хост localhost, порт **15432**, БД `homepilot`, пользователь/пароль `postgres`/`postgres`.
- **API:** http://localhost:8001, документация http://localhost:8001/docs.

Остановка: `Ctrl+C`, затем `docker compose down`. Данные БД — в volume `homepilot_pgdata`, фото — в `backend/uploads`.

### Вариант 2: Локально

**БД:** PostgreSQL 14+, создать БД `homepilot`.

**Backend:**

```bash
cd backend
python -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt
cp .env.example .env
# Заполнить DATABASE_URL, DATABASE_URL_SYNC, SECRET_KEY
alembic upgrade head
python -m app.db.seed
uvicorn app.main:app --reload --host 0.0.0.0 --port 8001
```

**Frontend:**

```bash
cd frontend
npm install
npm run dev
```

Приложение: **http://localhost:3003**. Запросы к `/api` проксируются на http://localhost:8001 (см. vite.config.ts).

| Метод | Путь | Описание |

## 14. Бизнес-логика

- **Создание подписки:** клиент создаёт подписку (status=draft), переходит к оплате. После успешной оплаты — status=active, заполняются started_at, ends_at, создаются визиты на первый месяц по preferred_days (4 или 8 в зависимости от тарифа).
- **Продление:** при автопродлении (или следующем платеже) создаются визиты на следующий месяц; при паузе визиты на период паузы не создаются, ends_at сдвигается.
- **Перенос визита:** проверка «не позднее чем за 24 ч» до визита; лимит бесплатных переносов с уведомлением &lt;24 ч (правило уточняется в продукте).
- **Неявка (no_show):** исполнитель отмечает no_show — визит списывается с пакета, не переносится.
- **Подтверждение email:** после регистрации отправляется 6-значный код (SMTP или вывод в логи). Код вводится на странице /auth/confirm-email.
- **Оплата (тестовый режим):** после POST create-intent в логах бэкенда выводится `>>> PAYMENT: код подтверждения списания ...`. Клиент вводит этот код в POST /payments/confirm (payment_id, code).

## 17. Безопасность и соответствие

- **Пароли:** хэширование bcrypt.
- **JWT:** access (короткий срок) + refresh; проверка роли на каждом защищённом эндпоинте.
- **HTTPS** в продакшене.
- **ПДн (РК):** сбор только необходимых данных; политика конфиденциальности; согласие при регистрации; возможность экспорта/удаления данных (заложить в политику).
- **Загрузка файлов:** только изображения; проверка типа и размера; сохранение под непредсказуемым именем; раздача через backend или signed URL.

## 18. Типичные проблемы и решение

| Проблема | Решение |
|----------|---------|
| Код подтверждения email не приходит | Без SMTP код выводится в терминал (uvicorn или `docker compose logs -f backend`). Искать строку `>>> КОД ПОДТВЕРЖДЕНИЯ`. |
| Оплата не проходит | В тестовом режиме после create-intent взять код из логов (`>>> PAYMENT: код подтверждения списания`) и ввести в POST /payments/confirm. |
| CORS при запросах с фронта | Проверить, что origin фронта (например http://localhost:3003) в CORS_ORIGINS в .env. |
| БД недоступна | Проверить DATABASE_URL (порт 15432 для Docker). Убедиться, что контейнер БД запущен и миграции применены. |
| Фото не открываются | Фото отдаются с backend по /uploads/...; проверить UPLOAD_DIR и права на каталог. |
| Очистить все подписки (тест) | `cd backend && python scripts/clear_subscriptions.py` (в активированном venv). |

## 19. Глоссарий

| Термин | Определение |
|--------|-------------|
| **Слот бытовой помощи** | Один запланированный выезд исполнителя к клиенту на выполнение задач по чек-листу (визит). |
| **Визит (Visit)** | Сущность в БД: дата, время, статус, привязка к подписке и исполнителю, чек-лист и фото. |
| **Чек-лист** | Набор пунктов (ChecklistItem) по шаблону (ChecklistTemplate) для типа уборки (лёгкая/полная). |
| **Подписка (Subscription)** | Оформленный тариф клиента: тариф, тип квартиры, адрес, дни и время, статус (draft/active/paused/cancelled). |
| **Тариф** | Пакет: количество визитов в месяц (4 или 8), тип уборки (лёгкая/полная), опции (бельё, растения, глажка — для Премиум). |
| **Закреплённый исполнитель** | executor_id в подписке; по возможности одному клиенту назначается один и тот же исполнитель. |
| **No-show** | Неявка клиента; визит отмечается исполнителем, списывается с пакета. |
