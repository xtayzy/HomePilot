# Техническое задание: HomePilot (фронтенд и бэкенд)

Полное ТЗ на разработку веб-приложения и серверной части сервиса подписки на бытовые задачи для рынка Алматы (с перспективой масштабирования).

**Связанные документы:** [Бизнес-модель](BUSINESS_MODEL_KZ.md), [Тарифы Алматы](TARIFFS_ALMATY.md). **ТЗ на бэкенд (FastAPI):** [BACKEND_TZ_FASTAPI.md](BACKEND_TZ_FASTAPI.md).

---

## 1. Общее описание проекта

### 1.1 Назначение

HomePilot — платформа подписки на бытовую жизнь: уборка, вынос мусора, закуп продуктов и хозяйственных товаров, выгул животных, мелкий бытовой ремонт и другие стандартные задачи по дому. Клиент выбирает тариф, указывает тип квартиры и расписание **слотов бытовой помощи**; в каждом слоте он собирает нужные задачи из доступного списка. Платформа назначает исполнителя, ведёт календарь слотов/визитов, приём оплаты и фото-отчёты. Администратор/оператор управляет пулом исполнителей, расписанием и поддержкой.

### 1.2 Цели разработки

- **Клиент:** выбор тарифа, оформление подписки, оплата, календарь слотов/визитов, выбор задач внутри слота, перенос/пауза, просмотр фото-отчётов, обращение в поддержку.
- **Исполнитель:** список визитов и задач на день/неделю, чек-лист по визиту/задачам, загрузка фото-отчёта, связь с клиентом/поддержкой.
- **Администратор:** клиенты, подписки, исполнители, визиты, финансы, поддержка (чаты/тикеты), настройки тарифов и городов.

### 1.3 Границы проекта (MVP)

- Один город: **Алматы**.
- Тарифы: Старт, Базовый, Оптимум, Комфорт, Премиум — уровни подписки на бытовые задачи (слоты бытовой помощи; состав и лимиты по задачам — по документу тарифов).
- Типы квартир: студия, 1-, 2-, 3-комн.
- Частота слотов/визитов: 4 или 8 в месяц.
- Языки интерфейса: русский, казахский (i18n с первого релиза).
- Платформа: веб-приложение (адаптивное; мобильное приложение — отдельный этап, в ТЗ не входит).

### 1.4 Вне границ MVP (после запуска)

- Мобильные приложения (iOS/Android).
- Специализированные отдельные пакеты (например, только закуп без уборки, большие ремонты, уход за детьми и т.п.) поверх базовой подписки.
- Второй город (Астана) — расширение справочников и зон.
- Реферальная программа и промокоды (можно заложить в модель данных, но не реализовывать в MVP).

## 2. Роли пользователей

| Роль | Описание | Доступ |
|------|----------|--------|
| **Клиент (Client)** | Покупатель подписки. Регистрируется, выбирает тариф, оплачивает, управляет визитами, смотрит отчёты. | Веб-приложение (клиентский кабинет). |
| **Исполнитель (Executor)** | Сотрудник/подрядчик, выполняет визиты. Получает задания, отмечает выполнение, загружает фото. | Веб-приложение (кабинет исполнителя) или упрощённый мобильный веб. |
| **Администратор (Admin)** | Сотрудник HomePilot. Управляет контентом, пользователями, визитами, тарифами. | Админ-панель (веб). |
| **Оператор поддержки (Support)** | Отвечает на обращения клиентов и исполнителей. Может просматривать заказы и чаты. | Админ-панель или отдельный раздел с ограниченными правами. |

*Оператор может быть подролью Admin с ограничением на разделы (только тикеты/чаты и просмотр карточки клиента/визита).*

## 3. Функциональные требования

### 3.1 Регистрация и авторизация

| ID | Требование | Комментарий |
|----|-------------|-------------|
| REF-1 | Города: код, название (ru/kk), активен ли для продаж. | Пока один — Алматы. |
| REF-2 | Тарифы: код (start, basic, optimum, comfort, premium), название, уровень пакета (lite/standard/premium и т.п.), кол-во слотов в месяц, рекомендуемая длительность слота, настройки по категориям задач (доступные категории: уборка, мусор, закуп, выгул, мелкий ремонт, поручения; лимиты по задачам). | Данные из TARIFFS_ALMATY. |

| REF-6 | Категории задач: код (cleaning, trash, shopping, pet_walk, minor_repair, errands и др.), названия (ru/kk), дефолтная длительность, флаги риска (например, требуется дополнительное согласие для определённых типов работ). | Для UI и бизнес-правил по задачам. |

### 3.3 Клиентский сценарий

| ID | Требование | Детали |
|----|-------------|--------|
| C-1 | Лендинг/главная | Описание сервиса, тарифы (название, что входит, цена «от»), кнопка «Начать» → регистрация или выбор тарифа. |
| C-2 | Выбор тарифа | Список тарифов (Старт, Базовый, Оптимум, Комфорт, Премиум); при выборе — выбор типа квартиры → отображение итоговой цены в месяц. |
| C-3 | Оформление подписки | После выбора тарифа и квартиры: адрес (город, улица, дом, квартира, подъезд, этаж, код домофона и т.п.), контактный телефон, предпочтительные дни и окно времени (например, вторник 10:00–14:00). Для Премиум — опции: замена белья, полив цветов, глажка. Подтверждение оферты (чекбокс). |
| C-4 | Оплата | Переход на оплату: сумма за первый месяц в тенге; способ — банковская карта (эквайринг). После успешной оплаты — подписка активна, создаются визиты на текущий месяц по выбранным дням. |
| C-5 | Личный кабинет клиента | Дашборд: текущий тариф, дата следующего списания, ближайший визит. Разделы: «Мои визиты», «Подписка и оплата», «Мой адрес и настройки», «Поддержка». |
| C-6 | Календарь визитов | Список визитов (дата, время, статус: запланирован, выполнен, отменён, перенесён). Для каждого визита — исполнитель (имя, фото), адрес, возможность переноса (если разрешено правилами: за 24 ч). |
| C-7 | Фото-отчёт по визиту | После выполнения визита — просмотр фото по пунктам чек-листа (галерея). Кнопка «Есть претензия» → форма/чат в поддержку. |
| C-8 | Перенос визита | Выбор новой даты/времени из доступных слотов; не более 1 бесплатного переноса с уведомлением <24 ч в месяц (правило из тарифов). Иначе визит считается состоявшимся. |
| C-9 | Пауза подписки | Кнопка «Поставить на паузу»: выбор дат начала и окончания (макс. 2 недели подряд, ограничение по количеству раз в год — по оферте). В период паузы визиты не планируются, срок подписки сдвигается. |
| C-10 | Отмена подписки | «Отменить подписку» — доступ до конца оплаченного периода; автопродление отключается. |
| C-11 | Профиль | Редактирование: ФИО, телефон, email. Смена пароля. |
| C-12 | Поддержка | Форма обращения (тема, текст) или чат. История обращений. При создании претензии по визиту — привязка к визиту. |

## 4. Бэкенд

### 4.1 Стек (рекомендации)

- **Язык:** Node.js (TypeScript) или Python (FastAPI/Django).
- **Фреймворк API:** REST; при необходимости позже — WebSocket для чата.
- **БД:** PostgreSQL (основные данные, транзакции, отчёты).
- **Кэш/очереди:** Redis — сессии, кэш, очередь задач (напоминания, письма).
- **Файлы:** Хранение фото отчётов — S3-совместимое хранилище (или локальное хранилище с путём в конфиге); ссылки в БД.
- **Платежи:** Интеграция с платёжным провайдером РК (например, Kaspi Pay, CloudPayments, или другой эквайринг) — приём платежей, webhook для подтверждения.
- **Почта:** SMTP или сервис (SendGrid, Mailgun и т.п.) для писем подтверждения и уведомлений.

### 4.2 Архитектура

- Монолит с чётким разделением на модули (users, subscriptions, visits, payments, support, notifications). При росте — выделение сервисов (payments, notifications) в отдельные процессы.
- Все запросы к API (кроме публичных: лендинг, оферта) — с JWT в заголовке. Роль проверяется на каждом защищённом эндпоинте.

### 4.3 Модели данных (основные сущности)

> **Примечание по этапам.** Для MVP допускается использовать существующую «клининговую» схему (tariffs с `cleaning_type` и `visits_per_month`, чек-листы только по уборке, без отдельных `task_categories` и `visit_tasks`). Расширенные сущности и поля (слоты, категории задач, `visit_tasks`, `task_options`) ниже описаны как целевая модель **после запуска MVP** и не являются обязательными для первой версии.

- **users** — id, email, password_hash, role (client | executor | admin | support), name, phone, locale (ru/kk), created_at, updated_at. Для executor: photo_url, zones (JSON или отдельная таблица), status (active/blocked).
- **cities** — id, code, name_ru, name_kk, is_active.
- **apartment_types** — id, code (studio, 1room, …), name_ru, name_kk, duration_light_min, duration_full_min.
- **tariffs** — id, code (start, basic, …), name_ru, name_kk, slots_per_month, default_slot_duration_min, allowed_task_categories (JSON: список кодов задач и лимитов по ним), is_active. (можно внедрять поэтапно; для MVP достаточно старых полей cleaning_type/visits_per_month).
- **tariff_prices** — tariff_id, apartment_type_id, price_month_kzt (integer, тенге).
- **task_categories** — id, code (cleaning, trash, shopping, pet_walk, minor_repair, errands и др.), name_ru, name_kk, default_duration_min, is_active. (опционально, после MVP; для старта категории можно зашить в чек-листы и UI без отдельной таблицы).
- **checklist_templates** — id, task_category_id или общий шаблон, apartment_type_id (опционально), sort_order. checklist_items — id, template_id, title_ru, title_kk, sort_order.
- **subscriptions** — id, user_id (client), tariff_id, apartment_type_id, city_id, address (JSON: street, building, flat, entrance, floor, doorcode, comment), preferred_days (JSON: [1,5] для пн, пт), time_slot_start, time_slot_end, task_options (JSON: дополнительные настройки по категориям задач, например лимиты на ремонт/закуп), status (active, paused, cancelled), started_at, ends_at, paused_until (nullable), executor_id (закреплённый исполнитель), created_at, updated_at. (для MVP можно начать без `task_options` и хранить пожелания по задачам в комментариях к визиту).
- **visits** — id, subscription_id, executor_id (nullable до назначения), scheduled_date, time_slot_start, time_slot_end, status (scheduled, in_progress, completed, cancelled, no_show, rescheduled), completed_at, client_rating (опционально), created_at, updated_at. Правило: визиты (слоты бытовой помощи) генерируются при активации подписки и при продлении месяца.
- **visit_tasks** — id, visit_id, task_category_id, payload (JSON с деталями задачи: список покупок, комментарий по ремонту, маршрут выгула и т.п.), planned_duration_min, status (planned, in_progress, completed, cancelled). (расширение после MVP; на старте можно обойтись чек-листами и общим комментарием к визиту).
- **visit_photos** — id, visit_id, checklist_item_id (nullable), file_path (S3 или относительный путь), uploaded_at.
- **visit_checklist_results** — visit_id, checklist_item_id, done (boolean), photo_id (nullable).
- **payments** — id, user_id, subscription_id, amount_kzt, currency, status (pending, completed, failed, refunded), external_id (id в платёжной системе), paid_at, created_at.
- **support_tickets** — id, user_id (client/executor), visit_id (nullable), subject, status (open, in_progress, closed), created_at, updated_at.
- **support_messages** — id, ticket_id, author_id (user), author_role, body, created_at.
- **executor_zones** — executor_id, city_id, zone_name или area (район) — для привязки исполнителя к районам.
- **subscription_pause** — id, subscription_id, paused_from, paused_to, reason (optional) — история пауз.

Связи: у подписки много визитов; у визита много фото и записей чек-листа. Платежи привязаны к user и subscription. Тикеты — к user и опционально к visit.

### 4.4 API (основные группы эндпоинтов)

**Публичные (без авторизации):**
- `GET /api/cities` — список активных городов.
- `GET /api/tariffs` — тарифы с ценами по типам квартир (для лендинга и выбора).
- `POST /api/auth/register` — регистрация клиента.
- `POST /api/auth/login` — вход (email, password) → access + refresh token.
- `POST /api/auth/forgot-password` — запрос на сброс пароля.
- `GET /api/legal/offer` — текст оферты (по локали).

**Клиент (роль client):**
- `GET /api/me`, `PATCH /api/me` — профиль.
- `POST /api/subscriptions` — создание подписки (тариф, тип квартиры, адрес, дни, окно времени, опции); после этого переход на оплату.
- `GET /api/subscriptions/current` — текущая подписка.
- `PATCH /api/subscriptions/:id` — изменение адреса/дней/времени; пауза; отмена.
- `GET /api/visits` — мои визиты (фильтр по периоду, статусу).
- `GET /api/visits/:id` — детали визита + фото-отчёт если выполнен.
- `POST /api/visits/:id/reschedule` — перенос визита (новая дата/время); бэкенд проверяет правила (24 ч, лимит переносов).
- `GET /api/payments` — мои платежи.
- `POST /api/payments/create-intent` — создание намерения оплаты (сумма, subscription_id) → возврат client_secret или ссылка на платёжную страницу (зависит от провайдера).
- `POST /api/support/tickets`, `GET /api/support/tickets`, `GET /api/support/tickets/:id/messages`, `POST /api/support/tickets/:id/messages` — тикеты и сообщения.
- `POST /api/visits/:id/complaint` — создать претензию по визиту (привязка к тикету или создание тикета).

**Исполнитель (роль executor):**
- `GET /api/executor/visits` — мои визиты (по дням).
- `GET /api/executor/visits/:id` — детали визита + чек-лист.
- `POST /api/executor/visits/:id/start` — начать визит.
- `POST /api/executor/visits/:id/complete` — завершить визит (тело: массив { checklist_item_id, done, photo_file? } или multipart с фото).
- `POST /api/executor/visits/:id/upload-photo` — загрузка фото к пункту чек-листа (если отдельно от complete).
- `GET /api/executor/profile`, `PATCH /api/executor/profile` — профиль и зоны.

**Админ (роль admin/support):**
- CRUD по клиентам, подпискам, визитам, исполнителям (списки и карточки).
- `POST /api/admin/visits/:id/assign-executor` — назначить исполнителя.
- `PATCH /api/admin/visits/:id` — перенос, отмена.
- CRUD тарифы, цены, чек-листы, города, типы квартир.
- `GET /api/admin/support/tickets`, ответы в тикетах.
- `GET /api/admin/payments` — список платежей.
- Приглашение исполнителя: `POST /api/admin/executors/invite` → возврат ссылки или кода.

**Webhook:**
- `POST /api/webhooks/payment` — уведомление от платёжного провайдера об успешной оплате; обновление payment и продление подписки, создание визитов на новый период при автопродлении.

### 4.5 Бизнес-логика на бэкенде

- При первой оплате подписки: создать визиты на первый месяц по preferred_days (не более 4 или 8 в зависимости от тарифа).
- При продлении (автосписание): создать визиты на следующий месяц; при паузе — не создавать визиты на период паузы, сдвинуть ends_at.
- Правило переноса: проверка 24 ч до визита; счётчик «коротких» переносов в месяц (например, 1 бесплатный).
- Неявка: исполнитель отмечает «no_show» — визит списывается с пакета (не переносится).
- Закрепление исполнителя: при назначении визита предзаполнять executor_id из subscription.executor_id; при смене исполнителя в админке — обновлять subscription.

## 5. Фронтенд

### 5.1 Стек (рекомендации)

- **Фреймворк:** React (TypeScript) или Next.js (SSR для лендинга и SEO).
- **Роутинг:** React Router или встроенный в Next.js.
- **Состояние:** React Query (TanStack Query) для запросов к API; для глобального UI/пользователя — Context или Zustand.
- **Формы:** React Hook Form + Zod (или Yup) для валидации.
- **UI:** компонентная библиотека (например, Material UI, Chakra UI, или собственный набор на Tailwind CSS). Адаптивная вёрстка (mobile-first).
- **i18n:** react-i18next (или next-i18next) — русский и казахский; ключи по разделам (auth, tariffs, dashboard, support и т.д.).
- **Сборка:** Vite (React) или Next.js CLI.

### 5.2 Структура приложения (три входа)

1. **Публичный сайт + онбординг** — один домен/путь: лендинг, тарифы, оферта, регистрация/вход, выбор тарифа и оформление подписки до оплаты.
2. **Клиентский кабинет** — после входа (роль client): дашборд, визиты, подписка, профиль, поддержка.
3. **Кабинет исполнителя** — после входа (роль executor): визиты на день/неделю, выполнение визита (чек-лист + фото).
4. **Админ-панель** — отдельный путь (например, /admin) или поддомен; вход по роли admin/support.

Роутинг можно разделить: `/` — лендинг; `/app/*` — клиент; `/executor/*` — исполнитель; `/admin/*` — админка. Или один SPA с редиректами по роли после логина.

### 5.3 Страницы и экраны

**Публичные:**
- Главная (лендинг): блоки о сервисе, тарифы (карточки с ценой «от»), CTA «Начать» / «Выбрать тариф».
- Тарифы: детальное описание каждого тарифа (текст из ТЗ/тарифного документа); выбор «Подключить» → переход к регистрации или к оформлению (если уже авторизован).
- Оферта: статическая страница (контент из API или markdown).
- Вход / Регистрация / Восстановление пароля.
- Оформление подписки (после выбора тарифа и типа квартиры): шаги — адрес, контакты, дни и время, опции (Премиум), подтверждение оферты, переход к оплате.
- Оплата: страница с формой карты (или редирект на платёжный виджет/страницу провайдера); после успеха — редирект в кабинет и сообщение «Подписка активна».

**Клиентский кабинет:**
- Дашборд: текущий тариф, следующее списание, ближайший визит (дата, исполнитель, адрес); быстрые ссылки на визиты и поддержку.
- Мои визиты: список/календарь; фильтр по периоду. Карточка визита: дата, время, статус, исполнитель; кнопки «Перенести», «Фото-отчёт» (если выполнен).
- Фото-отчёт: галерея по пунктам чек-листа; кнопка «Есть претензия».
- Подписка и оплата: текущий тариф, тип квартиры, адрес, дни; кнопки «Изменить адрес/время», «Пауза», «Отменить подписку». История платежей.
- Профиль: ФИО, телефон, email, смена пароля, язык интерфейса.
- Поддержка: список обращений; форма нового обращения; чат по выбранному тикету.

**Кабинет исполнителя:**
- Визиты на сегодня / на неделю: список карточек (время, адрес, клиент, тип уборки).
- Детали визита: адрес (и ссылка на карту), контакт, чек-лист. Кнопки «Начать», «Завершить».
- Экран выполнения: по каждому пункту чек-листа — галочка + загрузка фото. В конце — «Завершить визит».
- Профиль: ФИО, телефон, фото, зоны работы (только просмотр или редактирование — по политике).

**Админ-панель:**
- Дашборд: метрики (подписки, визиты, платежи, тикеты).
- Клиенты: таблица, поиск, открытие карточки (профиль, подписка, визиты, платежи, тикеты).
- Подписки: таблица; создание/редактирование; привязка исполнителя.
- Визиты: календарь или таблица; фильтры; назначение исполнителя, перенос, отмена.
- Исполнители: таблица; приглашение; блокировка.
- Тарифы и цены: форма редактирования тарифов и сетки цен; чек-листы по тарифам.
- Справочники: города, типы квартир.
- Поддержка: список тикетов, ответы в чате по тикету.
- Платежи: список транзакций.

### 5.4 Компоненты (общие)

- **Layout:** шапка (лого, навигация по роли, язык, выход), подвал (оферта, контакты).
- **TariffCard:** карточка тарифа (название, описание, цена «от», кнопка).
- **VisitCard:** карточка визита (дата, время, статус, исполнитель, действия).
- **PhotoReport:** галерея фото по пунктам чек-листа.
- **ChecklistForm:** чек-лист с галочками и полями загрузки фото (для исполнителя).
- **DataTable:** таблица с сортировкой и фильтрами (админка).

### 5.5 Адаптивность и доступность

- Все экраны корректно отображаются на мобильных (breakpoints: 320, 768, 1024).
- Кнопки и поля достаточного размера для тач-устройств.

## 6. Интеграции

- **Платежи:** API провайдера (создание платежа, webhook). Хранение только id транзакции и статуса; не хранить данные карты.
- **Email:** отправка писем подтверждения регистрации, сброс пароля, напоминание о визите (за день или в день визита). Шаблоны с поддержкой локали.
- **Карты (опционально):** отображение адреса визита для исполнителя (статическая карта или ссылка на Яндекс/Google Maps).

---

## 7. Безопасность и соответствие

- **Пароли:** хэширование (bcrypt или argon2).
- **JWT:** короткий срок жизни access-токена; refresh-токен в httpOnly cookie или в теле ответа с хранением на клиенте осторожно.
- **HTTPS** везде.
- **ПДн (РК):** сбор только необходимых данных; политика конфиденциальности; согласие пользователя при регистрации; возможность экспорта и удаления данных по запросу (опционально в MVP, но заложить в политику).
- **Роли:** на бэкенде проверка роли на каждом эндпоинте; клиент не имеет доступа к данным других клиентов и к админ-эндпоинтам.
- **Загрузка файлов:** только изображения; проверка типа и размера; сохранение под непредсказуемым именем; раздача через backend или через signed URL хранилища.

---

## 8. Нефункциональные требования

- **Доступность:** целевое время отклика API — до 500 ms для простых запросов в нормальных условиях.
- **Резервное копирование БД:** регулярные бэкапы (ежедневно минимум).
- **Логирование:** логи запросов и ошибок; логирование действий админа (кто что изменил) — желательно с первого релиза.
- **Мониторинг:** здоровье API (health check), при возможности — алерты при падении или росте ошибок.
